ndo-run · scoring ops reference

Score, tag & publish products — by asking Claude, not the droplet

A plain-English reference to everything the foodhealthco-ndo-ops plugin can do, what it won't let you do, and the guardrails that keep prod safe. Built for dietitians.

19 commands · dev-first by default · every write previewed before it runs

The whole thing in one breath: after a one-time setup, you tell Claude in plain English what you want — “re-tag and re-score these 5 products”. Claude picks the right command, shows you exactly what it's about to change, waits for your OK (and makes you type prod for live writes), runs it, then verifies the changes actually landed.

You ask in English→ Preflight preview→ You approve→ Run→ Postflight verify

No SSH, no droplet console, no manage.py to memorize. You never touch a command line — you just talk to Claude, and it drives the plugin for you.

Two ways to point at products: paste the IPM IDs (“…for IDs 3067665, 3067666”) or name a source (“…for source tyson_20260521”). If your intent is ambiguous, Claude asks before running anything.

02

Guardrails

What stops a mistake from reaching production. These are on by default — you don't enable them.

default Dev-first

Every new op starts on --target dev, a throwaway database. You move to prod only once dev looks right.

confirm Type-to-confirm prod

Any live write pauses and makes you literally type prod to continue. There is no silent prod write.

preview Preflight

Before any write, Claude buckets every input ID: ✓ update, ↻ skip, or ✗ block (a precondition failed). You see the damage before it happens.

verify Postflight

After a run, Claude re-queries the DB to confirm the writes landed. If it reports drift (“expected 5, observed 3”), something silently failed — screenshot it, that's a real bug.

safe Dry-run anything

Ask for a dry-run and Claude shows the exact command + preflight without writing a thing. Always safe, on any command.

limit Batch clamp

Batches above 1000 are clamped with a loud warning — a huge list can drop the scoring DB connection (ENG-965). Overridable only on purpose with --allow-large-batch.

sync Runs synchronously

Defaults to --sync true so results are real by the time the run finishes — no waiting on a background worker to pick up the job.

loud Propagation reminder

App search is built from approved scores, so it updates at approval — approve_scores --with-reindex. Skip the flag and Claude reminds you rather than leaving a silent gap; scoring alone won't reach search.

blocked HeroDB writes

--db platform is blocked pending ENG-897. It needs an explicit --force and still only does a single-DB swap. Leave it to eng.

audit Delete needs a paper trail

Destructive commands require a reason and operator note, and archive rather than hard-delete. Do dry-runs on dev first.

04

What you can & can't do solo

The full scoring → publish chain is yours. A few things route to engineering.

✓ You can run these yourself

  • Tag, impute & score — backfill_tags, backfill_imputation, backfill_fhs on a set of IDs or one source, dev or prod.
  • QA report — generate_qa_report, then review the xlsx before approving.
  • Approve scores — approve_scores, only once the QA review looks right.
  • Publish to clients — send_to_clients, the irreversible end of the chain.
  • Dry-run anything — always safe, no writes.

✗ Ask engineering for these

  • New source from scratch — creating IPM rows from raw vendor data (create_products / bulk_create_products).
  • Archive / delete — remove_products_and_scores, archive_table. Destructive.
  • Any credential error — InvalidAccessKeyId or “permission denied” after a retry.
  • Any postflight drift > 0 — writes that didn't land. Worth investigating.
05

A worked example

Re-tag and re-score 5 products you just edited in admin — the everyday recipe.

What you type into Claude

Just talk. In order:

  1. “Dry-run backfill_tags against dev for IPM IDs 3067665,3067666,3067667,3067668,3067669”
  2. “Run backfill_tags against prod for those IDs” → type prod to confirm
  3. “Dry-run backfill_fhs against dev for those same IDs”
  4. “Run backfill_fhs against prod for those IDs”
  5. “Generate a QA score report for those IDs”
What a preflight looks like
Pre-flight: backfill_tags
  Target: 🟢 NDO dev
  Input: 5 product_ids

Will WRITE:
  ✓  5  Will tag  (TaggingJob runs with overwrite=True)

Will SKIP (precondition fails):
  ✗  0  No ingredients_text
  ✗  0  Not in IPM

[dry-run] not executing
What a postflight looks like
Post-flight: backfill_tags
  ✅ all 5 expected writes landed
  source: ingestion_productmatch.updated_at >= 2026-05-22T14:15:32+00:00

If it says ⚠ drift — expected 5, observed 3, two products didn't get tagged. Screenshot and send to eng.

The under-the-hood command (Claude runs this for you — shown so you recognize it in output):

cd ~/Code/meltano-elt-pipelines
poetry run -- python .../ndo-run/scripts/ndo_run.py backfill_fhs \
  --ids 3067665,3067666 --target prod

The literal -- after poetry run matters — without it Poetry eats the --csv/--ids flag. Claude handles this; you never type it.

06

Glossary

IPM ID
“Ingestion Product Match” — a row in ingestion_productmatch. Its id is the number most commands want.
FHS
Food Health Score (0–100), computed by the FHS API from nutrients + ingredients + tags.
Source
The vendor + ingestion batch a product belongs to, e.g. tyson_20260521, nielsen, kroger. Lets you target a whole batch at once.
ScoringResult
A computed FHS awaiting approval — created by backfill_fhs.
ApprovedScoringResult
An approved score, ready to publish — created by approve_scores. This is what reaches clients.
Tagging (t2t)
Text2Tag: reads ingredient text and sets flags like is_deep_fried, is_artificial_colors. Rules live in t2t_v4.csv.
Preflight / Postflight
The preview before a write / the verification after it. Together they catch mistakes and silent failures.
Dry-run
Preview only — prints what would happen, writes nothing. Always safe.
dev / prod
Which database you're talking to. Dev = safe rehearsal. Prod = the live one.
07

Troubleshooting

“InvalidAccessKeyId” when uploading
Your DO Spaces keys in .env can't write to the btw-nutrition bucket (the backfills-test keys don't work). Ask eng for the prod-write keys (1Password: “DO Spaces — btw-nutrition prod write”).
Tagging says “100%” but no tags actually applied
Almost always DEFAULT_TAGGING_FILE isn't set to t2t_v4.csv in your .env. Watch for a WARN line “Tagging will run with NO RULES”. Match whatever prod currently uses.
Postflight shows drift
Some writes didn't land (“expected N, observed M”, M<N). Usual causes: the FHS API rejected items mid-batch for missing data (look for 4xx in the output), or a DB constraint. Screenshot the postflight and send to eng.
Preflight shows “✗ Not in IPM”
Those IDs don't exist on the target DB. Usual reasons: a typo, you're on dev but the IDs are from prod (dev has older data), or the product was archived. On dev this is normal — proceed.
“no preflight implementation for X”
That command doesn't have a preview/verify step yet — it still works, just without the safety preview. 12 of 19 commands have preflight today. Ask in #platform if you need one added.
Claude doesn't seem to use the plugin
In Claude Code run /plugin list — you should see foodhealthco-ndo-ops enabled. If not, reinstall and restart Claude Code.
