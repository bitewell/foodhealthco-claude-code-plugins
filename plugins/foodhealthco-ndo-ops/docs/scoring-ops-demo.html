The mental model

One loop, every time

You ask in plain English→ Preflight (what will change)→ You confirm→ It runs→ Postflight (did it land)
  • Preflight buckets every product: ✓ update / ↻ skip / ✗ block — before anything is written.
  • You confirm — on prod, Claude makes you type prod to proceed.
  • Postflight re-queries the DB and tells you if the writes actually landed (flags "drift" if not).

You're always driving. Claude never writes to prod without showing you the preflight and getting a typed confirmation first.

One-time setup · ~20–30 min

Get your machine ready

  1. Claude Code — install from claude.com/claude-code, sign in with @foodhealth.co.
  2. Repo access — confirm you're added to nutrition-data-ops and fhs-app with a working SSH key.
  3. Clone + deps — clone both repos under ~/Code, then poetry install in each.
  4. Install the plugin — /plugin marketplace add bitewell/foodhealthco-claude-code-plugins then /plugin install foodhealthco-ndo-ops@foodhealthco, restart.
  5. Drop in your .env — the scoped keys file (handed to you separately) at ~/Code/nutrition-data-ops/.env.
Verify with no risk: ask Claude for a --dry-run of backfill_fhs on dev for ids 1,2,3. You should see a preflight and the resolved command — and nothing executes.
Safety rails

Four habits that keep you safe

1 · Dry-run first

When unsure, ask for a dry-run. It shows the preflight and the exact command, writes nothing.

2 · Dev before prod

Rehearse on --target dev. Seeing ✗ Not in IPM on dev is normal — dev has different data.

3 · Typed prod confirm

Every prod write pauses for you to type prod. That pause is your last look at the preflight.

4 · Small batches

Drain backlogs a few hundred at a time. Anything over 1,000 is clamped with a warning.

Live demo · DEV only — nothing real changes

Let's run the loop together

  1. Confirm the plugin is on: /plugin list shows foodhealthco-ndo-ops.
  2. "Dry-run backfill_fhs against dev for IPM IDs 1,2,3." — read the preflight buckets; note nothing ran. (Dry-run only on dev — see below.)
  3. "Find a few dev IPM ids that have ingredients_text, then run backfill_tags against dev for them." — the real-write loop; watch the postflight verify it landed.
  4. "Score prod ID 3067619 and show me the result." — this is where a real FHS value appears. Dev can't produce one.
  5. Recap the rails: dry-run → dev → typed prod → batch limit.
Why scoring is dry-run-only on dev: backfill_fhs sends just the product ID; the FHS host looks up nutrients from its own (prod) data, so it doesn't recognize dev IDs — a dev scoring run returns 0. Rehearse the *mechanics* on dev; real scores only come back on prod, which is where CLI-869 runs (next slide).
CLI-869 · the real run, on prod

Protein powders, end to end

StepWhat you say to Claude
0 · Identify"Find protein powders not yet approved and export their ids."
1 · Tag (t2t)"Run backfill_tags against prod for those ids."
2 · Impute"Run backfill_imputation against prod for those ids." (adds missing macros incl. added sugars)
3 · Score"Run backfill_fhs against prod for those ids."
4 · QA"Generate a score QA report." → review the xlsx
5 · Approve"Build an approve CSV (product_id + fhs) and run approve_scores against prod."
6 · Publish"Run send_to_clients against prod for those ids."
If scoring 404s for every item, stop. The FHS scoring host is deprecated and mid-migration — a 404 means it's down, not transient. Don't retry blindly; escalate to eng. You can still generate a QA report (it scores locally), but approve + publish can't proceed until scoring is back.
Two things that trip people up

Know these before you run

Approve needs fhs too. approve_scores rejects an IDs-only CSV — it needs both product_id and fhs columns. Just ask Claude to "build the approve CSV from the ScoringResult rows" and it pulls the scores for you.
Some protein powders aren't scoreable yet. Anything still sitting in the ingestion table (not yet matched into a product record) won't show up in the identify query and can't be scored until it's bridged in. Flag those for eng — don't let them block the main batch.
Stop & ask eng if: postflight reports drift (expected N, got fewer), uploads fail with InvalidAccessKeyId, or anything wants an archive/delete. Screenshot and send.
Cheat sheet

What to remember

The loop

Ask → preflight → confirm → run → postflight. Always.

Order for CLI-869

identify → tag → impute → score → QA → approve → publish.

Safe by default

Dry-run when unsure. Dev before prod. Batches of a few hundred.

When in doubt

Just describe what you want in plain English — Claude picks the command and shows you before it writes.
